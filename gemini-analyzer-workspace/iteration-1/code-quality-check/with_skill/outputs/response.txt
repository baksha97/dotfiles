## ARCHITECTURE & DESIGN

### Strengths

1. **Clear Separation of Concerns**
   - Platform-specific logic cleanly separated (setup-macos.sh, setup-linux.sh, setup-alpine.sh)
   - Common shared logic in setup-common.sh
   - Single entrypoint (main.sh) provides unified interface

2. **GNU Stow Integration**
   - Excellent use of stow for symlink management
   - `--adopt` flag prevents data loss when adopting existing configs
   - Directory structure mirrors $HOME layout intuitively

3. **Profile-Based Configuration**
   - Personal/work profiles allow environment-specific customization
   - Git identity management is elegant and practical

4. **Backup Strategy**
   - Pre-existing configs are backed up before overwriting
   - Timestamped backup directories prevent data loss
   - Migration logic handles old directory layouts gracefully

### Weaknesses

1. **Inconsistent Shebang Usage**
   - `main.sh` uses `#!/bin/bash`
   - `setup-macos.sh` and `setup-common.sh` use `#!/bin/zsh`
   - `setup-linux.sh` uses `#!/bin/bash`
   - **Impact**: Potential portability issues; zsh-specific features may fail on systems without zsh

2. **Sourcing vs. Subprocess Execution**
   - Scripts are sourced (not executed), which means:
     - Variables and functions leak into the parent shell
     - `set -e` in sourced scripts behaves differently
     - Error handling is less predictable

## CODE QUALITY ISSUES

### 1. Error Handling Inconsistencies (MEDIUM PRIORITY)

**Location**: setup-linux.sh:56-65, setup-linux.sh:76-83

```bash
# Inconsistent error handling patterns:
LG_VERSION=$(curl -s "..." | jq -r .tag_name | sed 's/v//')
[[ -n "$LG_VERSION" ]] || { echo "Error: ..." >&2; exit 1 }
```

Some tools check for version fetch failures, others don't:
- ✅ lazygit, fzf, yq, docker-compose have version checks
- ❌ zoxide, just, uv, docker, tailscale, nodejs, vercel, opencode, claude lack version validation

**Recommendation**: Standardize version validation across all external tool installations.

### 2. Shell Compatibility Issues (HIGH PRIORITY)

**Location**: setup-linux.sh:33

```bash
mapfile -t pkgs < <(grep -v '^
s*#' meta/packages/linux.packages | grep -v '^
s*$')
```

The `mapfile` builtin is bash 4.0+ only. This will fail on:
- macOS default bash (3.2.57)
- POSIX sh environments

**Recommendation**: Use portable array construction or add bash version check.

### 3. Unquoted Variables (MEDIUM PRIORITY)

**Location**: Multiple files

```bash
# In setup-linux.sh:
$SUDO apt-get install -y "${pkgs[@]}"
# If $SUDO is empty and pkgs is empty, this fails

# In setup-common.sh:
for target in "${stow_targets[@]}"; do
  if [ -e "$target" ] && [ ! -L "$target" ]; then
```

While mostly quoted, edge cases around empty arrays could cause issues.

### 4. Hardcoded Values (LOW PRIORITY)

**Location**: setup-linux.sh:164

```bash
nerd_fonts_version="v3.3.0"
```

Font version is hardcoded. Consider:
- Fetching latest version dynamically
- Documenting why this specific version is pinned

## SECURITY CONSIDERATIONS

### 1. curl | bash Anti-pattern (MEDIUM PRIORITY)

Multiple instances of piping curl output directly to bash. While common in dotfiles, this:
- Executes code before verification
- Makes it hard to audit what ran
- Could be compromised if CDN is attacked

**Recommendation**:
- Download to temp file first
- Show user what will be executed
- Add opt-in confirmation for security-conscious users

### 2. Sudo Handling (LOW PRIORITY)

**Location**: setup-linux.sh:14-18

```bash
if [ "$EUID" -eq 0 ]; then
  SUDO=""
else
  SUDO="sudo"
fi
```

Good pattern for LXC/root containers, but consider:
- Validating sudo is available before using it
- Checking if user has passwordless sudo (which could be unexpected)

## PERFORMANCE CONSIDERATIONS

### 1. Redundant Font Cache Updates (LOW PRIORITY)

**Location**: setup-linux.sh:191-193

Font cache is only updated if fonts changed (good!), but `fc-cache -f` forces a full rebuild. Consider `fc-cache -fv` for verbose output or just `fc-cache` for incremental.

### 2. API Rate Limiting (MEDIUM PRIORITY)

Multiple GitHub API calls without authentication:
- lazygit version fetch
- fzf version fetch
- yq version fetch
- docker-compose version fetch

**Risk**: Unauthenticated GitHub API has rate limits (60 requests/hour/IP)

**Recommendation**:
- Add `GITHUB_TOKEN` support for authenticated requests
- Cache version information locally
- Add fallback to known good versions if API fails
